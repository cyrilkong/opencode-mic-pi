# AGENTS

_Generated by `scripts/generate-agents-doc.js`. Do not hand-edit._

## System Shape

- `mic` is the persistent intake window.
- `pi` is the execution foreman.
- `snap` remains a low-ceremony direct-action entry.
- All other router agents are backstage specialists.
- Runtime agent defaults come from `src/agent-catalog.js`.
- Mic presentation sources live under `src/presentation/mic-intake/`; the small barrel `src/presentation/mic-intake-spec.js` only re-exports them.
- Command presentation sources live under `src/presentation/commands/`; the small barrel `src/presentation/command-views.js` only re-exports them.
- Mic parser acceptance rules live separately in `src/intake-parser-contract.js` and `src/intake.js`.
- Prompt authoring sources live in `prompts/` and are bundled into `src/prompt-registry.js`.

## Runtime Defaults

- Public agents: `mic`, `pi`, `snap`
- Disabled builtin agents in managed mode: `plan`, `general`, `build`, `explore`

## Prompt Authoring Rules

- Keep each prompt split conceptually into role, mission, activation, boundaries, input contract, operating rules, and output contract.
- Do not duplicate Mic card shape, output UX style, or canonical intake samples inside prompt prose; keep those under `src/presentation/mic-intake/`.
- When changing visible Mic output shape/style, edit the presentation files first; only touch parser files when the machine-readable contract truly changes.
- Do not duplicate command output shape/style inside command logic; keep `/pi-dispatch`, `/pi-rematch*`, `/pi-up`, and `/pi-book` renderers under `src/presentation/commands/`.
- Do not duplicate router agent role summaries manually when a shared catalog fragment can generate them.
- After editing `prompts/`, `src/agent-catalog.js`, or Mic presentation sources under `src/presentation/mic-intake/`, regenerate bundled assets.

## Agent Directory

### `mic` · Mic

- Mode: `all`
- Surface: frontstage
- Prompt source: `prompts/mic.md`
- Description: User-facing intake mic and backlog reconciler. Captures messy input and distills it into a clean backlog, and can also reconcile backlog updates backstage for Pi.
- Mission: Turn messy user input or Pi-fed requirement drift into a stable, requirement-level backlog without taking over planning or execution.
- Cost posture: Always cost-sensitive. Default low-friction intake window.
- Best use cases: messy intake; scope shaping; requirement clarification; persistent backlog maintenance; backstage backlog reconciliation for Pi; language-stable front-window interaction
- Avoid when: implementation planning; agent routing; architecture decisions; deep execution reasoning
- Default output sections: `As-is`; `Task List`; `Questions`; `Ready For Dispatch?`
- Permission overrides: `{"question":"allow"}`

### `pi` · Pi

- Mode: `all`
- Surface: frontstage
- Prompt source: `prompts/pi.md`
- Description: Foreman and execution orchestrator. Takes a backlog and converts it into sequenced, delegated execution, whether Pi is frontstage or called backstage by Mic.
- Mission: Validate intake, package execution, delegate specialists, and keep progress aligned to the active backlog whether Pi is user-facing or acting as Mic's backstage control plane.
- Cost posture: Balanced orchestration spend. Optimize for fewer user-facing turns.
- Best use cases: execution orchestration; task packaging; worker delegation; blocker management; handoff and continuity management; backstage orchestration for Mic-frontstage sessions
- Avoid when: raw intake grooming; doing every specialist task itself; broad speculative redesign without evidence; specialist work that should be delegated
- Default output sections: `Status`; `What happened`; `Next step`

### `snap` · Snap

- Mode: `primary`
- Surface: frontstage
- Prompt source: `prompts/snap.md`
- Description: Fast direct-action agent for short, concrete, operational requests with minimal ceremony.
- Mission: Handle short operational tasks directly without dragging the user through orchestration ceremony.
- Cost posture: Low overhead. Prefer speed and directness.
- Best use cases: run a command; inspect a file; small config tweak; small direct fix; low-ceremony operational adjustment
- Avoid when: large multi-step engineering work; architecture review; extended debugging campaigns
- Default output sections: `Action`; `Result`; `Next Step`

### `co-pi` · Co-pi

- Mode: `subagent`
- Surface: backstage
- Prompt source: `prompts/co-pi.md`
- Description: Pi's second-brain for route checks, plan pressure-tests, task packaging, and process clarity.
- Mission: Pressure-test Pi's concrete proposal without replacing Pi or expanding into open-ended redesign.
- Cost posture: Selective advisory spend only when a concrete proposal exists.
- Best use cases: route sanity check; plan check; task packaging review; process drift pressure-test; second-opinion on a concrete proposal; breaking loops after repeated unsatisfied user feedback
- Avoid when: simple reversible steps; first-pass rediscovery; ceremonial reviews with no real proposal
- Default output sections: `Status`; `Verdict`; `Summary`; `Key Findings`; `Suggested Next Step`
